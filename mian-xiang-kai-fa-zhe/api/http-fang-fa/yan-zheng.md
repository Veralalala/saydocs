# 驗證

### 適用讀者

* 在應用或腳本中呼叫 **私有 API** 的前端 / 後端開發者
* 需要理解 **與簽名頁搭配** 的整體流程的整合方

如果你還不了解如何用錢包為請求做 EIP-712 簽名，請先閱讀[簽名](qian-ming.md)。

***

### 核心概念

#### 不使用傳統帳號體系

SayFi **不要求**使用者名稱與密碼。使用者的身分標識是 **相容以太坊的地址**；敏感操作的授權透過 **EIP-712 結構化簽名** 完成，由 Gateway 驗簽後視為「該地址發起了本次請求」。

#### Private 与 Public 路由

| 路径形态                                    | 是否需 EIP-712                      |
| --------------------------------------- | -------------------------------- |
| `/v1/private/...`（及同类的版本化 `private` 前缀） | **需要**（在请求头中携带签名相关字段，见签名文档）      |
| `/v1/public/...`                        | **不需要** 经过 Gateway 的 EIP-712 中间件 |

#### Private 與 Public 路由

| 路徑型態                                    | 是否需要 EIP-712                     |
| --------------------------------------- | -------------------------------- |
| `/v1/private/...`（以及同類版本化 `private` 前綴） | **需要**，在請求標頭帶上簽名相關欄位，詳見簽名文件      |
| `/v1/public/...`                        | **不需要** 經過 Gateway 的 EIP-712 中介層 |

> **说明**\
> 未匹配到 Gateway 所代理的版本化公網 / 私網路徑時，可能返回 404。請以目前部署的路由與版本為準。

驗簽通過後，從簽名**恢復出的地址**必須與呼叫方在 `X-Address` 中宣告的地址一致，請求才會通過。

***

### Gateway 內的驗權順序

對命中 **private** 且需要驗簽的請求，Gateway 在驗證 EIP-712 之前或過程中，可能依序執行以下檢查：

1. **黑名單**：可按地址，或「地址 + 客戶端 IP」組合攔截。
2. **限流**：按地址維度限流，降低濫用風險。
3. **EIP-712 校驗與防重放**：包含時間窗、`deadline`、簽名恢復與 `(地址, 時間戳)` 消費等，詳見 **EIP-712 請求簽名**。

***

### 錯誤回應

驗權與驗簽失敗時，HTTP 層會使用 **4xx/429** 等狀態碼，回應體為統一 JSON 結構，例如 `success: false`。基於安全考量，回應**不會**返回可協助攻擊的細粒度原因；完整碼表與實作對照請見[錯誤回應](cuo-wu-hui-ying.md)。

常見的驗證相關 `code` 包括以下示例，請以實際環境為準：

| `code`              | 含義（摘要）                                | 常見 HTTP 狀態 |
| ------------------- | ------------------------------------- | ---------- |
| `INVALID_SIGNATURE` | 標頭缺失 / 非法、鏈 ID 不在允許清單、簽名類型與路徑不符、驗簽失敗等 | 401 或 400  |
| `EXPIRED_REQUEST`   | 超出時間窗，或 `deadline` 已過期                | 401        |
| `TIMESTAMP_REUSED`  | 同一地址重複使用已消耗的時間戳                       | 401        |
| `AUTH_RATE_LIMITED` | 觸發按地址限流                               | 429        |
| `AUTH_BLACKLISTED`  | 命中黑名單                                 | 403        |
